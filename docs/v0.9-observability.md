# v0.9 pseudonymization observability

This note defines the v0.9 public-readiness contract for observing Gaze pseudonymization behavior from `gaze-lens`. It is intentionally limited to PII pseudonymization outcomes: detector ambiguity, validator vetoes, collision and anchor resolution, locale-aware fallback, and future daemon transport context.

It does not add MCP tools, CLI subcommands, or Rust implementation in this amendment.

## Surface

The locked public surface remains:

- MCP: `query`, `schema`, `list_tables`, `log_tail`, `log_grep`
- CLI: `serve`, `init`, `query`, `replay`, `check`, `demo`

Operators should observe v0.9 telemetry through existing profiles first:

- use a DB profile plus `query` for an operator-owned observability table or view;
- use a log profile plus `log_tail` / `log_grep` for process or app telemetry;
- use `replay` when the operator needs local raw-value verification.

A future dedicated observability command or MCP tool is not part of v0.9. It requires a later SPEC amendment with evidence that existing DB/log access is insufficient.

## Event shape

Observability events are token-safe operational metadata. They may include:

- `lens_session_id`, call id, profile id, source class, tool name;
- token id, token family, detector family, rule family, policy version;
- confidence bucket or coarse score band, never raw score evidence;
- locale, locale fallback reason, requested detector set, selected detector set;
- validator-veto reason code and non-sensitive shape metadata;
- anchor id, resolver strategy, collision-family id, and counts;
- daemon transport/source id and queue timing when v2 SDK ingest exists.

Events must not include:

- raw span text;
- raw surrounding context;
- rejected raw candidates;
- reversible hashes of raw PII;
- enough structured fragments to reconstruct the original value.

## Required cases

### Ambiguity side-channel

When recognizers disagree or a candidate is near a policy threshold, the event can record candidate kind families, selected kind, coarse confidence, and locale. This is a side-channel for policy tuning and replay diagnosis. It must not become a raw-evidence channel.

### Validator-veto

When a validator rejects a candidate, the event can record validator family and reason code, such as checksum failure, impossible date, unsupported number plan, or policy-disabled class. The rejected candidate stays out of the event.

### Collision family and anchor resolution

When token families collide or an anchor resolver chooses a canonical entity token, the event can record token family, collision family, anchor id, resolver strategy, and counts. Raw values remain recoverable only through operator replay.

### Locale-aware safety-net dispatch

Locale dispatch can record requested locale, fallback locale/detector set, reason code, and outcome counts. Unsupported or uncertain locale handling must prefer redaction over pass-through.

### Daemon relevance

The v2 SDK-ingest daemon must use the same event model. Daemon-specific transport metadata is allowed, but the daemon must still route retrieval through the shared session, redaction, manifest, and replay boundaries.

## Persistence rules

If events are persisted in the gaze-lens manifest, schema changes must be additive and backward-compatible: nullable columns or side tables keyed by session/call ids. Older binaries must be able to open the manifest and ignore observability additions.

Observability write failures must not return raw output. Retrieval remains fail-closed under the same contract as existing manifest writes.
